# Лабораторна робота №5
## Active Record: міграції, запити, scopes

**Курс:** Ruby on Rails
**Балів:** 15
**Попередня робота:** Лабораторна №4

---

## Мета

Навчитися працювати з міграціями для еволюції схеми бази даних, писати запити через Active Record та організовувати логіку запитів у моделі через scopes.

---

## Загальне завдання

Розширити додаток з Лабораторної №4 — додати нові поля та модель через міграції, реалізувати запити та scopes. Предметна область та основна модель — ті самі, що й у попередніх роботах (свій варіант).

## Частина 1 — Міграції

### Нове поле у основній моделі

Додати нове поле до існуючої таблиці основної моделі через міграцію (поле та його тип вказані у розділі "Варіанти"). Наприклад, для варіанту 1:

```bash
rails generate migration AddNotesToTasks notes:text
rails db:migrate
```

Оновити форму (partial `_form.html.erb`) та шаблон `show.html.erb` — додати нове поле. У звіті пояснити різницю між `string` і `text` у контексті бази даних.

### Нова додаткова модель

Створити нову модель для зберігання дочірніх сутностей, пов'язаних з основною моделлю (назва моделі та поля — у розділі "Варіанти"). Наприклад, для варіанту 1:

```bash
rails generate model Attachment filename:string url:string task_id:integer
rails db:migrate
```

Реалізувати повноцінний CRUD для нової моделі (контролер + маршрути + шаблони). Записи дочірньої моделі відображаються на сторінці `show` основної моделі.

> Зверніть увагу: поле `<main_model>_id` поки що просто число — зв'язок між моделями буде реалізовано у Лабораторній №6.

## Частина 2 — Запити через Active Record

Реалізувати у контролері або моделі наступні запити. Кожен запит продемонструвати на окремій сторінці або у консолі (`rails console`) зі скріншотом у звіті:

- Всі "неактивні" записи основної моделі, відсортовані за датою (`where` + `order`)
- Три найближчі за датою записи (`limit`)
- Записи зі специфічним значенням enum (наприклад, високий пріоритет / активний статус — див. варіант)
- Записи що містять певне слово у назві (`where` з `LIKE`)

Конкретизація для кожного варіанту — у розділі "Варіанти" нижче.

---

## Частина 3 — Scopes

Додати 4 scopes у основну модель. Загальні вимоги:
- два scopes по enum (по одному на два різних значення)
- один scope по числовому полю або boolean
- один scope по даті ("цього тижня", "у наступні N днів", "за останні N днів" — залежно від варіанту)

Приклад одного scope для варіанту 1:

```ruby
class Task < ApplicationRecord
  enum priority: { low: 0, medium: 1, high: 2 }

  scope :completed, -> { where(completed: true) }

  # решта scopes реалізуються за тим самим патерном
end
```

Список scopes для свого варіанту (опис, не готовий код) — у розділі "Варіанти" нижче. Назву методу і реалізацію (range, порівняння, лямбда з аргументом) студент обирає сам.

Використати scopes у контролері замість прямих запитів де це можливо.

---

## Частина 4 — Кастомна сторінка з scope

Додати нову сторінку, яка відображає результат одного з scopes. Для варіанту 1 це `/tasks/due_this_week` — завдання з дедлайном на поточному тижні. Для інших варіантів — див. розділ "Варіанти".

Сторінка має використовувати scope з моделі, не прямий запит у контролері.

Додати посилання на цю сторінку у навігацію (layout).

---

## Вимоги до реалізації

- Всі міграції застосовані, `schema.rb` актуальний
- Нова дочірня модель має повноцінний CRUD
- Запити реалізовані через Active Record, без сирого SQL
- Scopes визначені у моделі, а не у контролері
- Нова сторінка працює коректно і використовує scope

## Питання для захисту

1. Чим відрізняється `string` від `text` у міграції? Коли використовувати кожен?
2. Що таке `schema.rb` і чому його не можна редагувати вручну?
3. Чим відрізняється scope від звичайного методу у моделі?
4. Що таке `rails console` і як його використовувати для перевірки запитів?
5. Як переглянути який SQL генерує Active Record запит? Наведіть приклад для одного з запитів у вашому додатку.
